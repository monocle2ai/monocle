# Design: Piggyback server spans on the HTTP response

**Date:** 2026-07-21
**Status:** Approved (brainstorming), pending implementation plan

## Goal

Let a local `HttpRunner` test collect the **server-side** spans generated by an
HTTP request directly from the HTTP response, eliminating the Okahu cloud
round-trip for offline trace assertions.

Today, validating server-generated spans for an HTTP agent requires exporting
them to Okahu and fetching them back (`HttpRunner.get_remote_traces_source() ->
"okahu"`). This feature returns those spans in-band on the response so the test
framework can assert against them locally, with no cloud dependency.

## Activation (safe-by-default, two conditions)

Both must hold for any behavior change; otherwise the feature is completely
inert (no buffering, no trailer, no response mutation):

1. **Server opt-in:** `MONOCLE_ENABLE_TRACE_RETURN=true` in the server process.
   Without it, production servers never buffer or leak internal spans even if a
   client sends the trigger header.
2. **Client request:** the client sends request header
   `x-monocle-retrieve-traces: true`.

## Components

### A. Server — span collection (`apptrace`)

- New exporter `TraceReturnSpanExporter(MonocleInMemorySpanExporter)` that stores
  a finished span **only if** it carries the retrieve-traces scope attribute
  `scope.monocle_trace_return`. It is registered once at `setup_monocle_telemetry()` time
  **only when `MONOCLE_ENABLE_TRACE_RETURN=true`**. Because it filters on the
  scope attribute, the buffer stays empty unless a request has opted in — so it
  is bounded and does not retain spans for ordinary traffic.
- In the shared `extract_http_headers()` (the common header-processing entry used
  by every server framework): when env-on **and** the reserved header is present,
  call `set_scope("monocle_trace_return")` **before route execution**. The
  existing scope→attribute machinery then stamps `scope.monocle_trace_return` on
  every child span emitted during the request, which is exactly what the exporter
  filters on.

### B. Server — response trailer (`HttpSpanHandler` + per-framework wiring)

Shared logic lives on `HttpSpanHandler` so all frameworks reuse it:

- `collect_spans(trace_id)` — pull **and evict** the current trace's spans from
  `TraceReturnSpanExporter` (evict by matching `trace_id`, so concurrent requests
  never see each other's spans).
- `serialize_spans(spans)` — serialize `ReadableSpan`s using the **file-exporter
  span→dict JSON format** (the wire format `apptrace` already owns and that
  `test_tools`' `JSONSpanLoader` already deserializes). This serialization is
  factored into a shared helper reused by both the file exporter and this path.
- `build_trailer(spans)` — produce `<delimiter><gzip+base64(JSON)>` and set the
  response header `x-monocle-traces: v1; delim=<uuid4>`. The delimiter is a
  per-response uuid carried in the header, keeping the header tiny (no size
  ceiling) and making collision with real body bytes effectively impossible.

Per-framework response wiring (all HTTP server frameworks: `fastapi`, `flask`,
`aiohttp`, `lambdafunc`, `azfunc`) appends the trailer **after** the real body:

- **Streaming responses** are already chunked (no `Content-Length`). FastAPI's
  existing `_wrap_body_iterator` yields every real chunk untouched, then yields
  **one final trailer chunk**. Streaming is preserved end-to-end.
- **Buffered responses** append the trailer and **recompute `Content-Length`**
  (the server has the full body at that point).

Timing: the trailer is built at end-of-response, so all completed child spans are
included.

### C. Client — strip & stash (`apptrace` requests handler)

In `RequestSpanHandler` post-processing: if the response has the
`x-monocle-traces` header, read the delimiter, split the buffered body at the
delimiter, restore the clean body onto the response object, gunzip+base64-decode
the trailer to raw span JSON, and `setattr(response, "_monocle_remote_spans",
<raw span JSON>)`. This keeps `apptrace` free of any dependency on `test_tools` —
it only strips and stashes; it does not deserialize into `ReadableSpan`.

### D. Client — ingest (`test_tools` HttpRunner)

- Rename the existing `HttpRunner` class to **`HttpOkahuRunner`** (it keeps the
  `AgentTypes.HTTP_WITH_OKAHU = "http_with_okahu"` registration and its
  `get_remote_traces_source() -> "okahu"` behavior).
- Add a new **`HttpRunner`** class for this feature, registered under a new
  `AgentTypes.HTTP = "http"`. After the request, it reads
  `response._monocle_remote_spans`, deserializes with `JSONSpanLoader`, and feeds
  the spans into the validator's `MonocleInMemorySpanExporter`. Its
  `get_remote_traces_source()` returns a local/in-memory source (no Okahu).

## Data flow

```
client req + header x-monocle-retrieve-traces: true
  └─> server extract_http_headers: env-on & header present?
        └─> set_scope("monocle_trace_return") (before route execution)
              └─> child spans stamped scope.monocle_trace_return
                    └─> TraceReturnSpanExporter buffers only those spans
  ... route executes, produces (possibly streaming) response ...
  server response end (HttpSpanHandler):
    collect_spans(trace_id)  # pull + evict this trace's spans
    serialize -> gzip+base64
    set header  x-monocle-traces: v1; delim=<uuid4>
    append <delim><payload> after real body
      (streaming: final chunk; buffered: append + recompute Content-Length)
client requests handler (RequestSpanHandler post-processing):
  header present? split body at delim, restore clean body,
  decode trailer, setattr(response, "_monocle_remote_spans", raw_json)
    └─> HttpRunner: JSONSpanLoader.parse(raw_json)
          -> validator.memory_exporter.export(spans)
```

## Naming summary

| Thing | Name |
|---|---|
| Reserved request header (trigger) | `x-monocle-retrieve-traces: true` |
| Response header (announce + delimiter) | `x-monocle-traces: v1; delim=<uuid4>` |
| Server env guard | `MONOCLE_ENABLE_TRACE_RETURN=true` |
| Retrieve-traces scope | `monocle_trace_return` (attr `scope.monocle_trace_return`) |
| Server exporter | `TraceReturnSpanExporter(MonocleInMemorySpanExporter)` |
| Existing runner (renamed) | `HttpOkahuRunner` → `AgentTypes.HTTP_WITH_OKAHU = "http_with_okahu"` |
| New runner (this feature) | `HttpRunner` → `AgentTypes.HTTP = "http"` |
| Response attribute for stashed spans | `response._monocle_remote_spans` |

## Known limitations (v1)

- The **root HTTP/workflow span** ends *after* the response is fully sent, so it
  is **not** included in the returned set — only completed child spans
  (inference / tool / agent). Client-side assertions that target the server's
  root span will not be satisfied via this path.
- Client-side stripping assumes the **buffered** `requests` response (the
  `HttpRunner` default). Streaming *consumption on the client side* is out of
  scope for v1.
- The delimiter/collision guarantee relies on the response header surviving to
  the client. Direct client→server test setups (no proxy) satisfy this; the
  feature is not designed for arbitrary intermediary proxies.

## Testing

**Unit**
- `TraceReturnSpanExporter` stores only scope-tagged spans; ignores others.
- Trailer build/parse round-trip (serialize → gzip+base64 → decode → deserialize).
- `Content-Length` recompute correctness for buffered responses.
- Body split correctness at the delimiter, including binary and edge-case bodies.
- Inert behavior: env var unset, or header absent → no exporter registered / no
  scope set / no trailer / body unchanged.

**Integration**
- A FastAPI app (streaming path) plus one non-streaming framework app, run under
  Monocle instrumentation and driven by the new `HttpRunner`, asserting that the
  server's inference/tool spans arrive in the validator's memory exporter **with
  no Okahu involved**.
- Verify `HttpOkahuRunner` (renamed) still works via `"http_with_okahu"`.
- Concurrency: two overlapping opted-in requests each receive only their own
  trace's spans (trace-id eviction).
